# ProjectMK Coding Convention

이 문서는 `Source/ProjectMK` 내부 코드 수정 시 가장 먼저 확인하는 기준 문서입니다.
새 코드 작성, 기존 코드 수정, 리팩터링, 주석 추가 시 모두 이 문서를 우선합니다.

## 1. 기본 원칙

- Unreal Engine 기본 네이밍과 매크로 스타일을 따른다.
- 기존 코드베이스에서 반복되는 패턴을 우선 존중하되, 자동 생성 플레이스홀더나 깨진 텍스트는 남기지 않는다.
- 한 번 수정하는 파일은 가능하면 이 문서 기준에 맞춰 같이 정리한다.

## 2. 인코딩과 줄 끝

- 소스 파일(`.h`, `.cpp`, `.cs`)과 문서(`.md`)는 모두 `UTF-8 BOM`을 사용한다.
- 줄 끝은 `CRLF`로 통일한다.
- 모든 텍스트 파일은 마지막 줄 끝에 개행(`final newline`)을 반드시 둔다.
- 줄 끝 공백(trailing whitespace)은 남기지 않는다.
- 한글 주석과 문자열 사용은 허용한다. 단, 인코딩이 깨지지 않도록 반드시 UTF-8 BOM 상태를 유지한다.

## 3. 들여쓰기와 공백

- C++/C# 소스의 기본 들여쓰기는 `Tab`을 사용한다.
- 탭 너비는 4를 기준으로 본다.
- 정렬 목적이 아니라면 들여쓰기에 스페이스를 섞지 않는다.
- 제어문, 함수, 클래스 정의의 중괄호는 현재 프로젝트처럼 줄바꿈 후 여는 방식으로 유지한다.
- 빈 줄은 논리 블록을 나누는 데만 사용하고, 연속 빈 줄은 만들지 않는다.

## 4. 파일 헤더와 include 규칙

- Unreal이 자동 생성한 `Fill out your copyright notice...` 주석은 남기지 않는다.
- 기존 `// LINK` 배너는 유지 가능하지만, 새 수정에서 배너를 손댈 경우 `// LINK` 한 형태만 사용한다.
- 헤더 파일은 `#pragma once`를 사용한다.
- 헤더의 include는 최소화하고, 가능하면 forward declaration을 우선한다.
- `*.generated.h` 는 항상 해당 헤더의 마지막 include로 둔다.
- `.cpp` 파일은 자신의 헤더를 첫 include로 둔다.
- include 블록은 대체로 다음 순서를 따른다.
  1. 자신의 헤더
  2. 엔진/플러그인 헤더
  3. 프로젝트 헤더

## 5. 네이밍

- 클래스, 구조체, 함수, enum, UPROPERTY 이름은 Unreal 스타일의 PascalCase를 사용한다.
- bool 값은 `b` 접두사를 사용한다.
- UObject 계열 타입은 Unreal 접두사(`U`, `A`, `F`, `E`, `I`)를 유지한다.
- 약어를 새로 만들기보다, 이미 프로젝트에 있는 이름을 재사용한다.

## 6. 구현 스타일

- UObject 포인터 유효성 검사는 현재 코드베이스처럼 `::IsValid(...)` 사용을 우선한다.
- UObject가 아닌 일반 포인터나 `Find()` 반환값 확인에는 `nullptr` 비교를 사용한다.
- 헤더에서는 가능한 한 구현 세부사항을 숨기고, `.cpp`로 내린다.
- 짧은 getter/setter 정도만 헤더 inline으로 두고, 나머지는 구현 파일로 분리한다.

## 7. 주석

- 주석은 "왜 필요한지"가 있을 때만 짧게 남긴다.
- 한국어 주석 사용 가능하다.
- TODO는 `// TODO: ...` 형식으로 작성한다.
- 의미 없는 임시 주석, 깨진 문자, 생성기 기본 주석은 남기지 않는다.

## 8. 이번 정리에서 고정한 항목

- `.editorconfig` 기준을 추가해 UTF-8 BOM / CRLF / final newline / trailing whitespace 제거를 고정했다.
- 혼합 들여쓰기, 깨진 한글 주석, placeholder 파일 헤더를 우선 정리 대상으로 본다.
- 이후 코드 수정 시 이 문서를 확인하고, 수정한 파일은 가능한 범위에서 함께 정리한다.
